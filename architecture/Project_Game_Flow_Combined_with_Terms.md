# 프로젝트 기획 전체 흐름 + 게임 프로젝트 예시 (통합본)

> **두 문서의 모든 내용을 합침**:  
> ① 단계별 설명 + 단계별 매핑 + 상위 범주 정의  
> ② “게임기획 → 개발 → 배포” 실제 예시 흐름  
> 영어 기술 용어는 **첫 등장 시 병기**합니다.

---

## 🔎 상위 범주 정의

- **Why** – 왜 이 프로젝트를 하는지, 가치·문제·목표
- **What** – 무엇을 만들어야 하는지, 기능·품질속성·도메인
- **How** – 어떻게 만들지, 아키텍처·모듈·기술·보안·운영 방식
- **Who/When** – 누가·언제, 조직·팀·일정
- **Verify** – 검증, 테스트·실측·품질 게이트
- **Improve** – 개선·운영, 데이터 기반 반복

## 📘 단계별 설명

### 1. 비전 (**Why**)

- **무엇:** 해결하려는 문제와 가치, 성공 KPI(Key Performance Indicator, 핵심 성과지표)
- **왜:** 모든 의사결정의 나침반
- **순서 이유:** 목적이 먼저 확정되어야 방향성이 유지됨

---

### 2. 요구사항 & 품질속성 (**What**)

- **무엇:** 기능뿐 아니라 성능·확장·보안 등 품질 기준
- **왜:** 아키텍처와 기술 선택의 기준
- **순서 이유:** 원하는 품질을 알지 못하면 설계가 방황

---

### 3. 도메인 분석 & 경계 설정(DDD, Domain-Driven Design) (**What**)

- **무엇:** 문제 영역을 경계로 구분, 공통 언어 통일
- **왜:** 대규모 팀 협업의 기반
- **순서 이유:** 경계 없이 모듈·데이터 설계 불가

---

### 4. MVP 스코프(Minimum Viable Product) 결정 (**What**)

- **무엇:** 처음 시장/유저 검증을 위한 최소 기능
- **왜:** 위험과 비용을 줄이고 학습 빠르게
- **순서 이유:** MVP 범위가 아키텍처 크기를 좌우

---

### 5. 아키텍처 선택 (**How**)

- **무엇:** 시스템 구조 스타일(Client‑Server, Event‑Driven, ECS(Entity‑Component‑System: 엔티티‑컴포넌트‑시스템) 등)
- **왜:** 품질속성·운영환경을 만족하기 위해 필수
- **순서 이유:** 구조가 정해져야 모듈·기술 세부를 구체화 가능

---

### 6. 데이터 전략 (**How**)

- **무엇:** 데이터 모델, 저장소(RDB(Relational DataBase, 관계형 DB)/NoSQL(Not Only SQL, 비관계형)), 캐시, 일관성, 버전 관리
- **왜:** 장기 유지·성능·확장성의 핵심
- **순서 이유:** 아키텍처가 데이터를 중심으로 연결됨

---

### 7. 모듈 & API(Application Programming Interface) 설계 (**How**)

- **무엇:** 기능별 경계, API, 동기/비동기, 스레딩 모델
- **왜:** 팀별 병렬 개발·테스트·배포 가능
- **순서 이유:** 모듈이 확정되어야 세부 설계와 조직 구성 가능

---

### 8. 기술 스택 & ADR(Architecture Decision Record) (**How**)

- **무엇:** 엔진, 언어, DB, 빌드/테스트 툴, 결정 기록(ADR)
- **왜:** 실현 가능한 설계를 위해 필수
- **순서 이유:** 뒤늦게 바꾸면 모듈·데이터·API까지 영향

---

### 9. 보안 & 규정 준수 (**How**)

- **무엇:** 인증/인가, 데이터 보호, 감사 로그, 안티치트
- **왜:** 나중에 덧붙이면 비용 폭증
- **순서 이유:** 데이터와 API를 설계할 때부터 녹여야 함

---

### 10. 관측 가능성(Observability) (**How**)

- **무엇:** 로깅, 지표, 트레이싱, 경보, SLO/SLA/SLI(Service Level Objective/Agreement/Indicator)
- **왜:** 장애/성능 병목 조기 파악 필수
- **순서 이유:** 초기부터 계측 포인트를 넣어야 저비용

---

### 11. 테스트 전략 & 품질 게이트 (**Verify**)

- **무엇:** 단위·통합·리플레이·성능 테스트 및 합격 기준
- **왜:** 재작업 비용 방지, 품질 보증
- **순서 이유:** 모듈과 동시에 준비해야 효과적

---

### 12. DevOps & 배포 전략 (**How**)

- **무엇:** CI/CD(Continuous Integration/Continuous Delivery), IaC(Infrastructure as Code: 코드형 인프라), Blue‑Green(무중단 배포), Canary(카나리) 롤아웃/롤백
- **왜:** 빠른 반복과 안정적인 출시를 위해 필수
- **순서 이유:** 개발 초기부터 고려해야 혼란 방지

---

### 13. 팀 구조 설계 (**Who/When**)

- **무엇:** 모듈과 매칭된 팀 단위 및 협업 방식
- **왜:** 조직 구조가 아키텍처와 맞아야 병목 최소화
- **순서 이유:** 모듈 경계가 정해진 뒤 구성하는 것이 효율적

---

### 14. 스파이크/프로토타입 (**Verify**)

- **무엇:** 위험 높은 기술/성능 문제를 짧게 실험
- **왜:** 초기에 리스크 제거, 대규모 재작업 방지
- **순서 이유:** 본 개발 전 검증해야 설계 신뢰성 확보

---

### 15. 세부 설계 & 디자인 패턴 (**How**)

- **무엇:** 각 모듈 내부의 클래스/객체 수준 설계(Strategy, Observer, State, Command 등), FSM(Finite State Machine: 유한 상태 기계)
- **왜:** 재사용·유연성·테스트 용이성 향상
- **순서 이유:** 모듈 경계가 정해져야 내부 세부를 안정적으로 설계 가능

---

### 16. 로드맵 & 마일스톤 (**Who/When**)

- **무엇:** 릴리스 일정, 스코프 변경·리스크 관리
- **왜:** 진행 상황 추적 및 자원 관리
- **순서 이유:** 설계 기반이 잡힌 뒤 일정과 자원 현실적으로 배분 가능

---

### 17. MVP 구현 & 실측 검증 (**Verify**)

- **무엇:** MVP를 실제로 만들어 성능·품질·사용자 반응 측정
- **왜:** 설계와 가정이 현실에서 통하는지 확인
- **순서 이유:** 실행 후 개선 루프로 이어지도록 마지막 전 단계

---

### 18. 운영 & 개선 루프 (**Improve**)

- **무엇:** 운영 중 수집된 데이터 기반 지속 개선(A/B Test: 실험군/대조군 비교), APM(Application Performance Monitoring: 애플리케이션 성능 모니터링), GDPR(General Data Protection Regulation: 유럽 일반개인정보보호법) 등 준수
- **왜:** 장기 유지보수·경쟁력 유지
- **순서 이유:** 출시 이후에도 품질과 경험은 계속 진화해야 함

---

## 🎮 게임 프로젝트 실제 예시 흐름 (기획 → 개발 → 배포)

> 예시: 온라인 PvP(Player vs Player: 이용자 간 대결) 액션 게임

| 단계                      | 상위 범주    | 게임 프로젝트에서의 실제 흐름                                                                                                      |
| ----------------------- | -------- | --------------------------------------------------------------------------------------------------------------------- |
| **1. 비전**               | Why      | “간결한 조작과 빠른 매칭으로 즐기는 5v5 온라인 액션” 목표, 성공지표로 D30 리텐션·DAU(Daily Active Users, 일간 활성 사용자)·매치 대기시간 정의                      |
| **2. 요구사항 & 품질속성**      | What     | 동시접속 5만명, 120fps, 매치 대기 <10초, 글로벌 서버 필요 등                                                                             |
| **3. 도메인 분석 & 경계(DDD)** | What     | 전투, 경제(골드/상점), 소셜(길드/채팅), 메타(랭크/보상)로 구분                                                                               |
| **4. MVP 스코프**          | What     | 알파 버전: 전투+기본 매칭만 포함, 소셜/랭크는 이후 단계로 연기                                                                                 |
| **5. 아키텍처 선택**          | How      | 권위 서버 기반 Client‑Server, 이벤트 드리븐(Event‑Driven) 매치메이킹, ECS(Entity‑Component‑System) 게임로직                                |
| **6. 데이터 전략**           | How      | 플레이어 프로필은 NoSQL(Not Only SQL), 결제는 RDB(Relational DataBase), 인벤토리 캐시 전략 및 세이브 버전 관리                                   |
| **7. 모듈 & API 설계**      | How      | 전투 모듈, 매치메이킹 모듈, 인벤토리 모듈, 클라이언트 SDK(Software Development Kit, 소프트웨어 개발 키트) API 정의                                     |
| **8. 기술 스택 & ADR**      | How      | 언리얼 엔진5, Go 기반 매치 서버, AWS DynamoDB + S3(Simple Storage Service, 객체 스토리지) 선택, ADR(Architecture Decision Record)로 이유 기록 |
| **9. 보안 & 규정 준수**       | How      | 서버 권위 판정, 패킷 암호화, GDPR(일반개인정보보호법) 설계                                                                                  |
| **10. 관측 가능성**          | How      | 서버 지연/드롭률 지표, 인게임 크래시 리포팅, APM(Application Performance Monitoring) 도입                                                 |
| **11. 테스트 전략 & 품질 게이트** | Verify   | 전투 로직 단위테스트, 부하테스트, 리플레이 테스트                                                                                          |
| **12. DevOps & 배포 전략**  | How      | CI/CD 파이프라인, Blue‑Green(무중단) 롤아웃, CDN(Content Delivery Network, 콘텐츠 전송 네트워크) 통한 클라이언트 패치                              |
| **13. 팀 구조 설계**         | Who/When | 게임플레이 팀, 네트워크/백엔드 팀, 툴링 팀, QA(Quality Assurance, 품질 보증) 팀으로 구성                                                        |
| **14. 스파이크/프로토타입**      | Verify   | 대규모 매칭 성능 테스트용 프로토타입, ECS 성능 스파이크                                                                                     |
| **15. 세부 설계 & 디자인 패턴**  | How      | State/FSM, Observer, Strategy, Command 적용                                                                             |
| **16. 로드맵 & 마일스톤**      | Who/When | Q1: 알파, Q2: 클로즈드 베타, Q3: 런칭, 이후 시즌 업데이트 계획                                                                            |
| **17. MVP 구현 & 실측 검증**  | Verify   | 알파 운영 → 실제 지연·매치 대기시간 측정 → 개선점 도출                                                                                     |
| **18. 운영 & 개선 루프**      | Improve  | 런칭 후 A/B 테스트, 텔레메트리 기반 튜닝, 리팩토링 반복                                                                                    |

## 🚀 큰 흐름(기획 → 개발 → 배포) 매핑

- **게임기획 단계**  
  → (1) 비전 → (2) 요구사항 & 품질속성 → (3) 도메인 경계 → (4) MVP 스코프

- **개발 단계**  
  → (5) 아키텍처 → (6) 데이터 전략 → (7) 모듈/API → (8) 기술 스택 → (9) 보안  
  → (10) 관측 가능성 → (11) 테스트 전략 → (14) 스파이크 → (15) 디자인 패턴

- **배포 및 운영 단계**  
  → (12) DevOps → (13) 팀 구조 → (16) 로드맵 → (17) MVP 구현·실측 → (18) 운영·개선

---

## 📌 순서의 논리

- **Why → What → How → Who/When → Verify → Improve**  
- 앞 단계(전략)가 뒤 단계(설계·실행)를 제한 → 방향 일관성 유지  
- 위험 요소(성능·기술·보안)는 가능한 앞 단계에서 검증 → 비용 절감

---

## 📚 용어 사전(영어 → 의미)

- **API**: Application Programming Interface, 응용 프로그램 프로그래밍 인터페이스 — 모듈/서비스가 외부에 제공하는 호출 규약  
- **ADR**: Architecture Decision Record, 아키텍처 결정 기록 — 주요 기술/설계 의사결정과 근거를 남기는 문서  
- **CI/CD**: Continuous Integration / Continuous Delivery, 지속적 통합 / 지속적 배포 — 자동 빌드·테스트·배포 파이프라인  
- **DDD**: Domain‑Driven Design, 도메인 주도 설계 — 업무/게임 규칙 중심으로 모델·경계를 세우는 설계 접근  
- **MVP**: Minimum Viable Product, 최소기능제품 — 가장 작은 기능으로 가설을 검증하는 첫 제품  
- **ECS**: Entity‑Component‑System — 데이터(컴포넌트)와 처리(시스템)를 분리한 게임 아키텍처  
- **DOD**: Data‑Oriented Design, 데이터 지향 설계 — 캐시 효율을 위해 데이터 레이아웃을 중심으로 설계  
- **DSL**: Domain‑Specific Language, 도메인 특화 언어 — 특정 문제영역 전용의 간단한 언어  
- **P2P**: Peer‑to‑Peer — 중앙 서버 없이 참가자 간 직접 통신하는 네트워크 방식  
- **KPI**: Key Performance Indicator, 핵심 성과지표 — 성공 정도를 측정하는 지표  
- **SLO/SLA/SLI**: Service Level Objective/Agreement/Indicator — 서비스 품질 목표/계약/지표  
- **APM**: Application Performance Monitoring — 애플리케이션 성능 모니터링 도구/체계  
- **IaC**: Infrastructure as Code — 코드를 통해 인프라(서버/네트워크)를 정의·관리하는 방식  
- **CDN**: Content Delivery Network — 전 세계 엣지 노드로 콘텐츠를 빠르게 배포하는 네트워크  
- **GDPR**: General Data Protection Regulation — EU 일반개인정보보호법  
- **SDK**: Software Development Kit — 특정 플랫폼/서비스용 개발 도구 묶음  
- **FSM**: Finite State Machine — 유한 상태 기계, 상태 전이에 따라 동작하는 모델
