Part 5: Data Flow & Communication Patterns

Event-Driven Communication

Loosely-coupled systems communicate via EventBus

Publisher Example:

CombatSystem publishes CharacterDamagedEvent when damage occurs

Subscriber Example:

UI Manager subscribes to show damage text and update health bars

EventBus: Game/DragonicTactics/StateComponents/EventBus.h/cpp

Event Types: Game/DragonicTactics/Types/Events.h

EventBus::Instance().Subscribe<CharacterDamagedEvent>(
  [](const CharacterDamagedEvent& e) {
    // Handle event
  }
);

Direct Communication

Tightly-coupled systems pass references directly

Example:

PlayerInputHandler receives GridSystem* and CombatSystem* pointers

Reason:

Immediate gameplay operations require fast, direct access without event overhead

Implementation: Game/DragonicTactics/States/GamePlay.cpp:Update()

playerInputHandler.Update(
  gridSystem,
  combatSystem,
  turnManager
);

Data-Driven Architecture

Game content loaded from JSON files

Character Data:

Stats, abilities, spell slots defined in characters.json

Assets/Data/characters.json

Map Data:

Grid layout, tile types, spawn points in maps.json

Assets/Data/maps.json

Parser: External/json.hpp (nlohmann/json)

Registry: Game/DragonicTactics/StateComponents/DataRegistry.h/cpp

auto data = DataRegistry::Instance()
  .GetCharacterData("Dragon");

Factory Pattern

Centralized object creation

Purpose:

Single point of character instantiation, loads data from DataRegistry

Returns:

std::unique_ptr<Character> with full ownership semantics

Ownership:

Transferred to GameObjectManager for lifecycle management

Factory: Game/DragonicTactics/Factories/CharacterFactory.cpp:Create()

auto character = CharacterFactory::Create(
  CharacterTypes::Dragon,
  position
);
